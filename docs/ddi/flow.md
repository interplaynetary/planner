  1. Where buffers live: 
  Knowledge layer, not Plan    
  layer                      
                               
  BufferZone is stored in    
  BufferZoneStore (src/lib/know
  ledge/buffer-zones.ts). This
  is deliberate and correct. A
  buffer zone is standing
  configuration — it defines
  how much inventory to hold
  for a given
  ResourceSpecification at a
  given location, independent
  of any specific plan or
  order.

  The three VF layers map
  cleanly:
  Knowledge   → RecipeStore +
  BufferZoneStore  (what we
  know how to make, and where
  we buffer)
  Planning    → PlanStore +
  ReplenishmentSignal (what
  we've decided to do)
  Observation → Observer +
  EconomicResource    (what
  actually happened)

  ReplenishmentSignal is the
  boundary object — it's
  generated by DDMRP and lives
  in BufferZoneStore.signals,
  but when approved it crosses
  into PlanStore as a
  Commitment.

  ---
  2. Organization between
  buffers

  Buffers are connected through
   the recipe DAG. Between any
  two adjacent buffer points,
  there is a non-decoupled
  segment of processes. The
  downstream buffer replenishes
   from the upstream buffer (or
   raw material). Your codebase
   captures this with
  BufferZone.upstreamLocationId
   — the distribution hub's
  buffer points back to the
  manufacturing site's buffer.

  positioning.ts:rankDecoupling
  Candidates() expresses this
  as leverageScore — how many
  other recipes also consume
  that spec as input. A buffer
  with high leverage sits at a
  shared intermediate output
  and decouples many downstream
   flows at once (e.g., Welded
  Frame feeds every LifeTrac
  recipe).

  ---
  4. The Ch 7 iteration that
  loops back to Ch 6

  This is the key insight. Ch 7
   says: after you compute
  TOR/TOY/TOG for every buffer,
   sum the total inventory
  investment. If it's too
  large, you have too many or
  too widely-spaced decoupling
  points. If customer CLT is
  still too long, you need
  more.

  The iteration is:
  position → size → evaluate
  investment & CLT → reposition
   → re-size → ...

  In your codebase:
  rankDecouplingCandidates()
  returns candidates ranked by
  positioningScore. You pick
  some subset, run
  computeBufferZone() on each,
  inspect the resulting tog
  values (total inventory), and
   if the investment is wrong
  you adjust the candidate set
  and re-run. The loop
  terminates when the
  positioning/sizing tradeoff
  is acceptable.

  This is not automated in the
  current codebase — it's a
  human decision loop supported
   by the scoring functions.
  decouplingTests() in
  positioning.ts gives you the
  six pass/fail tests that Ch 6
   uses to validate each
  candidate once you've
  provisionally placed it.
